# AI Agent 学习笔记阅读地图与术语约定

1. 给整套笔记补一个统一的阅读入口  
让这 11 章内容不只是“并列的文章”，而是一条有先后层次的学习路径。

2. 统一后续写作时的术语口径  
避免同一套笔记里出现“概念差不多，但叫法不一致”的情况。

## 2. 整套笔记统一采用什么术语口径？

### 2.1 `AI Agent`、`Agent`、`智能体`

这三个词在整套笔记里默认指向同一个核心概念。

统一约定：

- **第一次引入一个主题时，优先写作 `AI Agent（智能体）`。**
- **在正文中大多数位置，优先简称为 `Agent`。**
- **`智能体` 作为中文解释性称呼保留，但尽量不和 `Agent` 做无意义混用。**

这样做的原因是：

- `AI Agent` 更适合第一次定义概念
- `Agent` 更适合贯穿全文，和工作流、单多 Agent、工具调用等表达更一致
- `智能体` 适合做中文解释，但如果频繁和 `Agent` 交替，阅读节奏会变散

### 2.2 `大模型` 与 `LLM`

统一约定：

- **第一次出现时优先写作 `大语言模型（LLM）`。**
- **后文大多用 `大模型` 或 `LLM`，但在同一小节内尽量保持一致。**

### 2.3 `工作流`、`Workflow Pattern`

统一约定：

- **正文主称呼统一使用 `工作流` 或 `工作流模式`。**
- **英文名只在第一次介绍某种模式时保留，例如 `ReAct`、`Plan-and-Execute`、`Router`、`Reflection`。**

### 2.4 `单 Agent` 与 `多 Agent`

统一约定：

- **整套笔记默认使用 `单 Agent`、`多 Agent`。**
- **`单智能体`、`多智能体` 只作为中文补充解释，不作为主标题口径。**

原因很简单：  
后续章节已经大量采用 `单 Agent / 多 Agent` 这组表达，统一后更顺。

### 2.5 记忆类型术语

统一约定：

- **短期记忆**：当前任务工作台，重点是连续推进
- **长期记忆**：跨任务仍然有价值的信息
- **情景记忆**：过去具体发生过的事件
- **语义记忆**：抽象出来的规则和知识
- **共享记忆**：多个 Agent 或多个流程共享的公共上下文

其中：

- **优先使用 `共享记忆`，不再单独使用 `共识记忆` 作为主叫法。**

### 2.6 `工具调用` 与 `RAG`

统一约定：

- **`工具调用` 解决“怎么和外部系统交互、怎么执行动作”。**
- **`RAG` 解决“怎么在回答和决策前拿到外部知识”。**

后续写作中，  
尽量避免把两者混成一个词义模糊的“大能力包”。

### 2.7 `评估` 与 `测试`

统一约定：

- **测试**：更偏系统是否按设计工作
- **评估**：更偏系统在真实任务里是否真的好用

这两个概念相关，但不完全等同。  
后续写作时尽量区分使用。

### 2.8 `安全`、`权限`、`人机协作`

统一约定：

- **安全**：关注错误动作、越权动作、不可控动作
- **权限**：关注系统边界和能力范围
- **人机协作**：关注哪些节点该让人参与确认、判断或接管

这三者是相互关联的，不建议拆开孤立理解。

### 2.9 `工程化` 与 `生产落地`

统一约定：

- **工程化**：更偏系统基础设施和运行能力
- **生产落地**：更偏真实环境中的可运营、可维护、可规模化

多数时候这两个词会一起出现，因为在 Agent 项目里它们高度相关。

## 3. 后续写新章节时，结构尽量保持什么风格？

为了让整套笔记读起来像一套课程，而不是一组松散文章，  
后续章节尽量延续下面这个结构节奏：

1. `## 1. 先说结论：...`  
先把这一章最核心的判断讲清楚。

2. 中间主体按“概念 -> 为什么重要 -> 怎么做 -> 常见误区 -> 落地顺序”展开  
不要求每章都机械一致，但整体节奏尽量接近。

3. 末尾保留小结  
让读者能用 3 到 6 个重点回收整章内容。

4. 尽量多回答“为什么”和“怎么判断”  
少写成只列名词、只下定义的说明文。

5. 尽量保留实用判断框架  
让读者读完后能拿它判断真实问题，而不只是记住一些术语。

## 5. 一句话总结这套笔记现在的主线

这套笔记现在的主线可以浓缩成一句话：

> Agent 不是“会聊天的大模型”，  
而是一个围绕目标持续感知、判断、行动，并在评估、安全和工程边界内稳定推进任务的系统。
